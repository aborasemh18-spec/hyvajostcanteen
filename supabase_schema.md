# Supabase PostgreSQL Database Schema Design (Canteen Management System)

This document details the database schema design for migrating the Angular Canteen Management System from Firebase Firestore to Supabase PostgreSQL (Phase 1). 

---

## Architectural Objectives & Rules Applied
1. **Preserve Business Logic**: The existing Angular services, components, and controllers are retained perfectly. Table column mappings align with TS interfaces.
2. **Snake Case Conversion**: Firestore document fields are mapped to PostgreSQL `snake_case` column names to align with standard SQL paradigms while retaining matching nomenclature.
3. **Primary Key Retentions**:
   - `employees.id` is maintained as a **BIGINT** matching the numeric identifier in the frontend.
   - `contractors.id` is maintained as a **BIGINT**.
   - `coupons.coupon_id` is maintained as **TEXT** to retain existing generated alphanumeric Firestore IDs or string keys.
   - `qr_cards.card_code` is mapped as **TEXT PRIMARY KEY**.
4. **Data Integrity & Relational Rules**: Explicit foreign key relations, constraints (`CHECK`, `UNIQUE`, `NOT NULL`), default values, and relational referential integrity (`ON DELETE CASCADE` / `SET NULL`) are implemented.
5. **Timestamping**: Every date-time field is specified as `TIMESTAMPTZ` (Timestamp with Time Zone) for internationalization and standardizing offset-handling.

```
       [Contractors] (business_name) <--- (contractor) [Employees] <----------+
             |                                                ^               |

```

## Table Schemas Detailed

### 1. Table: `contractors`
Stores contractor accounts, login credentials, and business metadata.

| Column Name | SQL Type | Nullable | Keys / Constraints | Description |
| :--- | :--- | :--- | :--- | :--- |
| `id` | `BIGINT` | `NOT NULL` | `PRIMARY KEY`, `GENERATED BY DEFAULT AS IDENTITY` | Numeric contractor ID. Auto-incrementing fallback but supports manual import values. |
| `name` | `TEXT` | `NOT NULL` | | Full name of the contractor contact person. |
| `business_name` | `TEXT` | `NOT NULL` | `UNIQUE` | Company/Business trade name. Referenced by employees. |
| `contractor_id` | `TEXT` | `NOT NULL` | `UNIQUE` | Secure alphanumeric Login ID used in authentication. |
| `password` | `TEXT` | `NOT NULL` | | Hashed/plain password for contractor panel. |
| `created_at` | `TIMESTAMPTZ`| `NOT NULL` | `DEFAULT NOW()` | Record creation timestamp. |

---

### 2. Table: `employees`
Stores company employees (both permanent and contractual).

| Column Name | SQL Type | Nullable | Keys / Constraints | Description |
| :--- | :--- | :--- | :--- | :--- |
| `id` | `BIGINT` | `NOT NULL` | `PRIMARY KEY` | Numeric employee ID mapped from frontend data. |
| `name` | `TEXT` | `NOT NULL` | | Employee's full name. |
| `email` | `TEXT` | `YES` | | Email address. |
| `employee_id` | `TEXT` | `NOT NULL` | `UNIQUE` | String badge number/code (e.g., "EMP001"). |
| `password` | `TEXT` | `NOT NULL` | | Authentication password. |
| `role` | `TEXT` | `NOT NULL` | `CHECK (role IN ('admin', 'contractual employee', 'employee', 'canteen manager'))` | Employee security and routing role. |
| `department` | `TEXT` | `YES` | | Department name. |
| `employee_category` | `TEXT` | `YES` | `CHECK (employee_category IN ('Staff', 'Technician'))` | Sub-classification. |
| `status` | `TEXT` | `NOT NULL` | `DEFAULT 'active'`, `CHECK (status IN ('active', 'deactivated'))` | Active status control. |
| `contractor` | `TEXT` | `YES` | `REFERENCES contractors(business_name) ON DELETE SET NULL` | Linked contractor business name if the employee is contractual. |
| `permanent_qr_code` | `TEXT` | `YES` | `UNIQUE` | Static QR string assigned to employee. |
| `assigned_qr_card` | `TEXT` | `YES` | `UNIQUE` | External physical card code string if associated. |
| `last_redeemed_date` | `TIMESTAMPTZ`| `YES` | | Last check-in scan or validation date. |
| `last_morning_breakfast_date`| `TIMESTAMPTZ`| `YES` | | Date-time of last morning breakfast redemption. |
| `last_lunch_date` | `TIMESTAMPTZ`| `YES` | | Date-time of last lunch redemption. |
| `last_evening_breakfast_date`| `TIMESTAMPTZ`| `YES` | | Date-time of last evening snack redemption. |
| `last_dinner_date` | `TIMESTAMPTZ`| `YES` | | Date-time of last dinner redemption. |
| `created_at` | `TIMESTAMPTZ`| `NOT NULL` | `DEFAULT NOW()` | Record creation date. |

### 3. Table: `coupons`

| Column Name | SQL Type | Nullable | Keys / Constraints | Description |
| :--- | :--- | :--- | :--- | :--- |
| `coupon_id` | `TEXT` | `NOT NULL` | `PRIMARY KEY` | Alphanumeric coupon key mapped from Firestore doc key. |
| `employee_id` | `BIGINT` | `YES` | `REFERENCES employees(id) ON DELETE CASCADE` | Associated employee owner (NULL for unassigned contractor pool). |
| `contractor_id` | `BIGINT` | `YES` | `REFERENCES contractors(id) ON DELETE CASCADE` | Associated contractor owner (NULL for standard corporate employees). |
| `date_issued` | `TIMESTAMPTZ`| `NOT NULL` | | Issuance date-time. |
| `status` | `TEXT` | `NOT NULL` | `DEFAULT 'issued'`, `CHECK (status IN ('issued', 'redeemed', 'expired'))` | Coupon state. |
| `redeem_date` | `TIMESTAMPTZ`| `YES` | | Timestamp of actual scan redemption. |
| `redemption_code` | `TEXT` | `NOT NULL` | `UNIQUE` | Secure randomized string used to perform scans. |
| `coupon_type` | `TEXT` | `NOT NULL` | `CHECK (coupon_type IN ('Breakfast', 'Lunch/Dinner', 'Snacks', 'Beverage'))` | Canteen food session type. |
| `slot` | `INTEGER` | `YES` | | Slot index code (calculated from type). |
| `is_guest_coupon` | `BOOLEAN` | `NOT NULL` | `DEFAULT FALSE` | Flag for guest-allocated passes. |
| `shared_by_employee_id`| `BIGINT` | `YES` | `REFERENCES employees(id) ON DELETE SET NULL` | Host employee ID who requested the guest pass. |
| `guest_name` | `TEXT` | `YES` | | Guest visitor's name. |
| `guest_company` | `TEXT` | `YES` | | Guest organization name. |
| `batch_id` | `TEXT` | `YES` | | ID of the batch generation loop. |
| `created_at` | `TIMESTAMPTZ`| `NOT NULL` | `DEFAULT NOW()` | DB receipt date. |

## Performance & Optimization Indexes
To maintain sub-millisecond query execution as the database size increases, we declare explicit B-Tree Indexes on the columns utilized by `WHERE`, `ORDER BY`, `JOIN`, or `FOREIGN KEY` loops:

1. **`employees`**:
   - `idx_employees_role` (for filtering menus, coupons, and admins)
   - `idx_employees_status` (excluding deactivated members during active loops)
   - `idx_employees_contractor` (speeding joins with parent contractor companies)
2. **`coupons`**:
   - `idx_coupons_employee_id` (retrieving personal coupon wallets)
   - `idx_coupons_contractor_id` (calculating unassigned pools)
   - `idx_coupons_status` / `idx_coupons_coupon_type` (filtering daily canteen screens)
   - `idx_coupons_batch_id` (removing/modifying coupon lots)
3. **`guest_coupon_requests`**:
   - `idx_guest_coupon_requests_employee_id` / `status` (employee request history)
4. **`pending_meal_requests`**:
   - `idx_pending_meal_requests_meal_date` (payroll processing and calendar scans)
5. **`notifications`**:
   - `idx_notifications_employee_id` + `is_read` (speeding unread inbox count indicators)
6. **`punch_events`**:
   - `idx_punch_events_created_at` (daily audits and terminal sync lists)
